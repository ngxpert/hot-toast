🔗 HTTP error interceptor

You can register hotToastHttpInterceptor on HttpClient to show an error toast when requests fail, then rethrow the error so your existing catchError / subscribe logic still runs. Use provideHotToastHttpInterceptor() to supply optional ignore rules and message overrides. On the server, no toast is opened.

Add the functional interceptor to the same provideHttpClient() call as your other features (for example withFetch()), and list hotToastHttpInterceptor in withInterceptors([...]) alongside your own interceptors.

Sample server test page — run npm run sample:http-test-server, then open that route while this app is on port 4200 (see environment.sampleHttpTestServerUrl).

Registration

Wire HttpClient and optional interceptor config at bootstrap (or in a route/environment injector that owns HTTP for that subtree).

Configuration

provideHotToastHttpInterceptor accepts a partial config object. All fields are optional; merge with defaults happens inside the provider.
